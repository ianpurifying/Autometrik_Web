1.1 Introduction 
Every teacher faces common challenges, such as determining whether their lessons 
are effective. After giving a test, teachers need answers to important questions. Are 
students learning what they should? Which topics need to be taught again? Were the test 
questions fair and useful? These questions cannot be answered by simply counting correct 
answers. Teachers need deeper analysis—examining patterns across the class, 
understanding which questions worked well, and identifying where students struggled. 
With proper assessment analysis, teachers can make smart decisions about what to do next 
in their classrooms. 
Classroom assessment plays a vital role in the Philippine K–12 curriculum. The 
Department of Education (DepEd) emphasizes using quality assessments to monitor 
student progress and enhance teaching effectiveness (DepEd, 2015). One cornerstone is the 
Mean Percentage Score (MPS)—the average of individual percentage scores—which 
provides a standardized measure to evaluate how well students have met learning targets 
(DepEd, n.d.; DepEd, 2022). When combined with item analysis, which measures item 
difficulty (percentage of students answering correctly) and item discrimination (ability to 
distinguish high from low performers), these tools enable systematic examination of test 
quality and significantly improve assessment reliability and validity (Brown, 2010). 
Item analysis helps teachers identify which questions function effectively and 
which need revision, supporting continuous test improvement (Santos & Reyes, 2023). 
MPS computation provides a clear metric that facilitates communication about student 
achievement across classes and grade levels. Together, these practices embody DepEd's 
commitment to evidence-based instruction—teaching informed by accurate data rather 
than intuition alone (DepEd, 2022). 
Recognizing these benefits, educators have increasingly adopted digital tools to 
support assessment practices. Web-based assessment systems provide centralized data 
storage, multi-device accessibility, real-time feedback, and integrated tools for scoring, 
analysis, and reporting (Gajardo & Balahadia, 2025). Teachers using such systems report 
high satisfaction due to the flexibility and immediate feedback that help them respond more 
effectively to student needs (Suhendar et al., 2020). 
The convergence of assessment best practices with modern web technologies 
creates an opportunity to support teachers more effectively than ever before. AutoMetrik 
is designed to realize this opportunity—providing a comprehensive, web-based platform 
that automates MPS computation, performs item analysis, manages quiz libraries, and 
generates actionable reports. By integrating these essential assessment functions into one 
cohesive system, AutoMetrik aims to help teachers harness the full power of assessment 
data, leading to improved test quality, more informed instructional decisions, and 
ultimately, better learning outcomes for students in the Philippine K–12 system.

1.2 Background of the Study 
Despite the well-established benefits of assessment analysis, significant barriers 
prevent many Philippine teachers from implementing these practices effectively. Filipino 
teachers often lack practical skills and training in item construction and analysis 
(Alcazaren, 2024), and assessment literacy—particularly in indigenous and underserved 
communities—remains inadequate, hampering the use of assessment data to inform 
instruction (Napanoy & Peckley, 2020). Even when teachers understand the theoretical 
importance of MPS computation and item analysis, practical classroom realities make 
consistent implementation difficult. 
Traditional pen-and-paper testing presents multiple operational challenges. Manual 
checking is time-consuming, especially for teachers handling multiple classes (Alruwais et 
al., 2018). This process is prone to human error—fatigue, distraction, and inconsistent 
scoring undermine assessment reliability (Fagbohun, 2021). Paper-based tests involve 
significant printing costs for budget-limited schools, particularly when multiple versions 
are needed for test security (Dermo, 2009). Physical papers can be lost, copied, or accessed 
inappropriately (Dendir & Maxwell, 2020). Results are delayed as teachers need days or 
weeks to complete checking, postponing crucial student feedback (Nguyen et al., 2016). 
Managing physical test papers creates storage and organizational burdens (Ripley, 2008). 
Beyond scoring mechanics, computational demands create additional barriers. Item 
analysis requires calculating difficulty and discrimination indices for each question—
processes involving sorting students, counting responses, and applying specific formulas 
(Brown, 2010). DepEd mandates these analyses for test quality and competency alignment 
14 
(DepEd, 2022), yet the calculations are time-intensive for teachers managing large classes 
and multiple subjects. Many teachers skip item analysis entirely or perform it superficially, 
missing opportunities to identify problematic items. Repeated reuse of un-analyzed 
questions gradually erodes test quality and masks curriculum gaps. Critical diagnostic 
data—such as unmastered skills or confusing distractors—go unnoticed or are reported too 
late, making instruction less responsive and data-driven. 
To address these burdens, many teachers have adopted digital tools. ZipGrade, a 
mobile scanning app commonly used in Philippine schools, allows teachers to scan paper 
answer sheets using phone cameras and provides automatic scoring with basic item 
statistics and individual MPS computation (Cortez et al., 2023; Suhendar et al., 2020). 
While ZipGrade reduces manual checking time and computes individual assessment MPS, 
it presents critical gaps limiting its Philippine context. 
First and most significantly, ZipGrade does not provide consolidated MPS across 
multiple assessments—critical for tracking class performance longitudinally and 
identifying learning trends over time, which DepEd requires under Department Order No. 
31, s. 2020 for comprehensive progress monitoring (DepEd, 2022). Teachers must export 
data from multiple assessments and manually calculate consolidated MPS in 
spreadsheets—reintroducing delays, error risks, and workflow disruption. Second, 
ZipGrade functions primarily as a scanning and scoring tool rather than a comprehensive 
assessment management system, lacking features for organizing quiz banks, tracking 
longitudinal performance, managing class rosters, or generating DepEd-formatted reports. 
Third, as a mobile-first application, ZipGrade's smartphone-optimized interface limits 
utility for tasks requiring larger screens, such as reviewing detailed analytics or preparing 
15 
comprehensive reports. Finally, ZipGrade's item analysis presents raw statistics without 
interpretive guidance or recommendations—teachers still need expertise to translate 
indices into actionable decisions. 
These limitations reveal a fundamental gap: no existing Philippine tool fully 
integrates quiz management, automated scoring, MPS computation, and item analysis into 
one complete, DepEd-compliant platform accessible to K-12 teachers. While international 
web-based assessment systems offer advanced features such as centralized storage, cross
device accessibility, and real-time analytics (Gajardo & Balahadia, 2025), these platforms 
are typically expensive, lack DepEd-specific compliance, or aren't designed for Philippine 
public school resource constraints. 
The gap between teacher needs and current tools necessitates AutoMetrik. Teachers 
need automated scoring and MPS computation based on DepEd standards, eliminating 
repetitive manual work. They need automatic item analysis with clear improvement 
recommendations. The system must manage the entire assessment lifecycle—quiz 
creation, administration, and result analysis—in one platform, generate DepEd-compliant 
reports for documentation and evaluation, and be accessible online on any standard device 
without specialized equipment. These needs reveal current tool limitations and the practical 
demand for fast, accurate, actionable data supporting classroom practice.  
AutoMetrik addresses these gaps by unifying all essential assessment functions into one 
platform tailored to the Philippine K-12 context. By automating time-intensive 
computational tasks, AutoMetrik makes evidence-based assessment practices practical and 
sustainable for teachers, ultimately supporting better instructional decisions and improved 
student learning outcomes.

General Objective 
This study develops and evaluates AutoMetrik, a web-based assessment system that 
addresses critical gaps in Philippine K-12 education. These gaps include time-consuming 
manual assessment processing, inaccessible item analysis tools, and the lack of integrated 
DepEd-compliant assessment management. AutoMetrik automates MPS computation, 
performs comprehensive item analysis with interpretive guidance, and unifies quiz 
management into one platform, providing teachers with timely, accurate, and actionable 
data to support evidence-based instructional decisions.

spreadsheet errors and scoring errors. 
1.2 Integrating quiz creation, administration, scoring, and analysis into a single

# AUTOMETRIK – SYSTEM ARCHITECTURE

---

## Security & Access Control

Autometrik uses a two-layer security model. First, Firebase Authentication verifies who a user is and assigns them a role — either Super Admin, Organization Admin, Teacher, or Student. Second, Firestore Security Rules enforce what each role is actually allowed to do, regardless of what the frontend shows.

Every user also has a status that must be **active** before they can do anything meaningful in the system. If a user is pending, suspended, or rejected, they are completely blocked from the entire platform. The frontend may hide certain UI elements based on role, but the real enforcement always happens on the server side.

Each role operates within strict boundaries. Super Admins manage organizations and platform-wide settings but are intentionally kept away from academic data like student submissions or class activity. Organization Admins can only see and manage their own organization — they cannot touch data belonging to another. Teachers can only access their own classes and the submissions tied to their deployments. Students can only see and interact with their own work.

---

## 1. Getting an Organization onto the Platform

When a organization wants to use Autometrik, they submit a registration form. This creates an organization record in the system with a **pending** status, and a corresponding admin account is created for them — also pending.

A Super Admin then reviews the registration. If approved, both the organization and its admin become fully active and can start using the platform. If rejected or left pending, the organization remains completely blocked.

---

## 2. Running the Organization

**Setting Up Academic Years**

Once an organization is active, the admin is guided through a one-time onboarding flow before reaching the main dashboard. Rather than dropping them into an empty screen, the system walks them through the minimum setup required to get the organization running — presented as a step-by-step checklist or progress stepper so they always know where they are and what's left.

The steps cover: creating the first school year, setting up at least one section, and noting the organization access code for teachers and students to register. Each step is completable inline without leaving the flow, and the admin can see a summary of what they've configured before finishing. Once all required steps are done, they land on the main dashboard with their organization already operational.

For returning admins starting a new school year, a lighter version of this flow is triggered — prompting them to activate the new year, review or carry over existing sections, and confirm the setup before proceeding. This avoids the blank-slate problem without forcing them through the full onboarding again.

Only one school year can be active at a time. When a new year is activated, the previous one is automatically archived. Archived years become read-only — no new sections, classes, or assessments can be created under them, but their reports remain accessible.

**Creating Sections**

The org admin creates sections (e.g., Grade 10 - Section A) tied to the active school year. Each section gets a unique access code that students use to join, and an adviser teacher is assigned to the section at the time of creation — giving the org admin a clear picture of who is responsible for each group. When a student tries to join, the system validates that the access code is correct and that both the student and the section belong to the same organization.

**Bringing in Teachers**

Teachers register using the organization's access code. They start with a pending status and must be approved by the org admin before they can access the platform.

**Bringing in Students**

Students register with their name, email, password, gender, and a section access code. Like teachers, they also start as pending and must be approved before they can access anything in the system.

---

## 3. Classes and Subjects

Once teachers are approved, they can create classes by linking a subject to a section within the active school year. Students are automatically associated with the class through their section enrollment — no manual addition needed.

A class can be active (fully functional) or archived (no new assessments, but reports are still visible).

---

## 4. Building Quizzes

Teachers build quizzes in quiz bank. Each quiz contains questions, topics, quantity of questions (10, 20, 30, 50, or 100 by default), and competency codes. All questions are multiple choice, with adjustable numbers of answer options per question.

By default, a quiz is private to the teacher who created it. A teacher can choose to share a quiz within their organization, making it available for other teachers in the same org to reuse. Quizzes are designed to be reusable — the same quiz can be deployed to multiple classes at different times.

---

## 5. Deploying an Assessment

When a teacher is ready to assess a class, they select a quiz on the quiz bank and a class, then configure how the assessment should run: when it opens and closes, how many attempts students get, whether questions are randomized, when scores are released, and which anti-cheat measures are active (detect tab-switching and disabled copy-paste).

The deployment moves through three states automatically: **scheduled** (not yet open), **active** (students can take it), and **closed** (no more submissions). These transitions are enforced by the system based on the configured time window — not just by the frontend.

---

## 6. The Assessment Experience

**What Students See**

Students only see assessments that are currently active, belong to a class they are enrolled in, and are within the allowed time window. Nothing else is visible to them.

**Starting the Exam**

When a student clicks "View Assessment", they are first shown a pre-exam preview screen displaying the assessment details — the title, topics covered, time limit, number of questions, and any attempt restrictions — so they can prepare before committing. Once they click "Start" from that screen, the system creates their submission record, starts their timer, and generates their randomized question order if randomization is enabled.

**During the Exam**

Answers are saved automatically at regular intervals and whenever the student moves between questions. If the student loses connection, accidentally closes the tab, or experiences any interruption, their answers are preserved locally on their device so nothing is lost — once they reconnect and return to the assessment, their progress is restored exactly where they left off.

**Submitting**

After the student reviews and confirms their submission, the system automatically grades their answers, computes their score, locks the submission so it can't be changed, and applies the teacher's configured rules for when and how scores are revealed.

---

## 7. Analytics and Insights

After submissions come in, teachers and org admins can access a full analytics layer. This includes overall class performance, score distributions, item analysis (how hard each question was, how well it discriminated between students), MPS (mean percentage score), Mean, Standard Deviation, competency mastery breakdowns, and rankings.

These calculations are done on demand and then cached — they don't recompute every time someone opens the dashboard, keeping the system fast and cost-efficient.

---

## 8. Reporting

**For Teachers:** Reports are scoped to their own classes. They can view item analysis, MPS, class performance summaries, and export data as PDF, CSV, or Excel.

**For Orgizations:** Reports span the entire organization — they can see school-wide dashboards, compare teacher and class performance, and pull consolidated reports for MPS, Item Analysis, and performance summaries. They cannot access data from any other organization.

---

## 9. Audit Logs

Every significant action in the system is permanently recorded and surfaced to the roles that need to see them. Teachers can view cheating-related events flagged during their deployments — tab switches, focus losses, and suspicious activity tied to their assessments — as well as quiz sharing events whenever one of their quizzes is made visible org-wide and class archiving actions for classes they own. Organization Admins have visibility into academic governance events within their org, such as quiz creation and edits, deployment changes, school year activations, user approvals or rejections, section management actions like creating or deleting sections, and access code usage — capturing every instance of a teacher or student joining the organization using a code. Super Admins have access to platform-level logs, including organization registrations, approval decisions, and login attempts. Every log entry captures who performed the action and their role at the time, what was affected and how it changed, when it happened down to the exact server-side timestamp, where it came from via IP address and device info, and whether it succeeded or failed — including the reason for any failure. Regardless of who views what, all logs are write-only and immutable — nothing in the system can modify or delete them.

---

## Platform Overview

Autometrik runs entirely serverless through Vercel, uses Firestore as the primary database with Firebase Realtime Database handling live exam monitoring, and manages authentication and role-based access control through Firebase Auth. The system is designed to support up to 1,000 concurrent users with strict data isolation between organizations — no school can ever see another school's data.

# Performance Tips (must implement)

## 1️⃣ Data Fetching & Caching

* **Cache Firestore results with TanStack Query** → reduces repeated reads and improves perceived speed.
* **Prefetch data for next pages** → smoother navigation and faster perceived load.
* **Scope onSnapshot listeners tightly** → listen only to the fields and documents a role actually needs, never the full collection.
* **Compute aggregations server-side via Next.js API routes or Firebase Cloud Functions** → offloads heavy calculations from the frontend.

## 2️⃣ Database Optimizations

* **Separate monitoring fields from answer payload in submissions** → teachers listen only to lightweight fields during live exams, not full answer data.
* **Use RTDB for ephemeral high-frequency writes** → student progress, current question, tab switch count during active exams.
* **Cache analytics results into a dedicated Firestore collection** → avoid recomputing on every dashboard open.
* **Paginate audit log queries** → only load the last 50–100 entries on open, fetch older entries on demand.

## 3️⃣ Frontend Optimizations

* **Lazy-load dashboard modules** → smaller initial bundle → faster first paint.
* **Memoize heavy components and charts** → prevents unnecessary re-renders.
* **Use React Suspense for async components** → reduces waterfall loading.
* **Leverage Next.js App Router server components** → reduce client-side JS where interactivity isn't needed.

## 4️⃣ Charts & Tables

* **Optimize chart rendering for large datasets** → consider canvas-based charts over pure SVG for performance.
* **Throttle or debounce RTDB and Firestore onSnapshot updates** → prevents render thrashing during high-frequency live exam monitoring.

## 5️⃣ Assets & Hosting

* **Keep static assets on Vercel CDN** → global fast delivery.
* **Optimize images** → compress and convert to WebP for faster load.
* **Use Next.js Image component** → automatic lazy loading and size optimization.

## 6️⃣ Monitoring & Maintenance

* **Run Next.js bundle analysis** → identify unused dependencies and reduce bundle size.
* **Profile frontend renders** → find bottlenecks in components or charts.
* **Monitor Firestore read/write counts** → audit log and onSnapshot listeners are the most likely sources of unexpected cost spikes.
